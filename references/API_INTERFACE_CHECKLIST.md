# API Interface Checklist

本清单用于设计、实现和审计 API / 接口变更。

目标：确保接口语义清晰、鉴权正确、错误一致、兼容可控、可测试。

---

## 1. When to Use

涉及以下情况时必须使用：

- 新增 API。
- 修改 API。
- 删除 API。
- 改变请求参数。
- 改变返回结构。
- 改变错误格式。
- 新增下载 / 导出接口。
- 新增后台管理接口。

---

## 2. Route and Naming

- [ ] 路由是否符合项目现有命名风格？
- [ ] HTTP method 是否语义正确？
- [ ] 路径是否稳定、清晰、不过度嵌套？
- [ ] 是否避免暴露内部实现名？
- [ ] 是否与已有接口重复？

---

## 3. Request Design

- [ ] 参数是否必要且明确？
- [ ] 参数类型是否校验？
- [ ] 必填和可选是否明确？
- [ ] 分页、排序、筛选是否有默认值？
- [ ] 是否限制 page_size / limit 最大值？
- [ ] 是否避免把敏感信息放在 URL query？

---

## 4. Response Design

- [ ] 返回结构是否符合项目约定？
- [ ] 是否只返回必要字段？
- [ ] 是否避免大响应？
- [ ] 列表接口是否包含分页信息？
- [ ] 空数据返回是否一致？
- [ ] 是否避免泄露内部字段？

---

## 5. Error Handling

- [ ] 错误格式是否符合项目约定？
- [ ] 400 / 401 / 403 / 404 / 409 / 500 是否区分清楚？
- [ ] 参数错误是否有可理解提示？
- [ ] 无权限是否返回正确状态？
- [ ] 资源不存在和无权限是否避免泄露信息？
- [ ] 外部服务失败是否被安全包装？

---

## 6. Auth and Permission

- [ ] 是否需要登录？
- [ ] 是否需要管理员权限？
- [ ] 是否检查资源归属？
- [ ] 列表接口是否按用户/租户过滤？
- [ ] 下载 / 导出是否鉴权？
- [ ] 是否避免只靠前端隐藏？

---

## 7. Compatibility

- [ ] 是否改变旧接口返回结构？
- [ ] 是否改变旧字段语义？
- [ ] 是否删除旧字段？
- [ ] 是否需要版本化？
- [ ] 是否需要迁移调用方？
- [ ] 是否有回滚方案？

---

## 8. Performance

- [ ] 是否分页？
- [ ] 是否避免大响应？
- [ ] 是否避免同步长任务？
- [ ] 是否有超时风险？
- [ ] 是否有 N+1 或慢查询风险？

---

## 9. Testing

- [ ] 正常请求。
- [ ] 缺失参数。
- [ ] 无效参数。
- [ ] 未登录。
- [ ] 无权限。
- [ ] 资源不存在。
- [ ] 空结果。
- [ ] 大数据 / 分页。

---

## 10. Interface Spec Format

建议接口文档包含：

```markdown
## API: [Name]

Method: GET / POST / PUT / DELETE
Path: /api/...
Auth: Required / Admin / Public

### Request

### Response

### Error Cases

### Permission Rules

### Compatibility Notes
```

---

## 11. Review Gate Questions

1. 这个接口是否和现有接口风格一致？
2. 权限是否在后端强制？
3. 错误格式是否一致？
4. 是否可能返回过多数据？
5. 是否破坏兼容性？
6. 是否有测试覆盖关键路径？
